# v17.17 — Conditional Placeholder (Do Not Implement Yet)

**Status:** PLACEHOLDER. Not ready for handoff. Do not start coding.

**Trigger to convert this into a real spec:** v17.15 and v17.16 are merged, the patient has used the app for **at least 1–2 weeks**, and telemetry dumps from real sessions have been reviewed.

The point of waiting is that v17.15's per-frame telemetry was built specifically to tell us *which* of three architectural changes is actually needed. Picking now, before data exists, is the failure mode this placeholder is designed to prevent.

---

## The three candidates

Exactly **one** of the following becomes v17.17. The choice is made from telemetry, not from intuition or vendor research.

### Candidate A — IPC transport rewrite (`MessageChannelMain` + preload)

**Pick this if** v17.15 telemetry shows:
- `frames[].dtMs` median > 50 ms or p95 > 100 ms (frame gaps dominate)
- `frames[].ageMs` median > 30 ms (gaze samples are stale by the time they reach the page)
- Frequent `trackingLost` events at the per-frame layer (not the tracker)
- Lag is the dominant patient complaint after v17.15/v17.16 ship

**Scope sketch (if picked):**
- Replace per-frame `view.webContents.executeJavaScript(...)` with `MessageChannelMain` + preload `MessagePort`
- Preload owns a last-sample buffer; a `requestAnimationFrame` loop consumes it
- Keep BrowserView; keep all v17.15/v17.16 algorithm work intact
- Estimated effort: 1–4 days; main risk is preload context-isolation regression

**Do NOT pick this if** target identity is the larger problem — IPC speed-ups don't fix wrong-target snapping.

### Candidate B — Semantic candidate registry

**Pick this if** v17.15 telemetry shows:
- High `targetSwitch` event rate (> 6/minute during normal browsing)
- High onset-cancel rate with `reason: 'winner_changed'` or `reason: 'candidate_moved'` (> 20% of onsets)
- Continued wrong-target clicks despite v17.15's Stable Sidebar patch
- Patient feedback: "still clicks the wrong thing" rather than "feels slow"

**Scope sketch (if picked):**
- Build a candidate registry from anchors / roles / accessible names / hrefs / visible rects
- `ytd-*` selectors become fallback evidence, not primary truth
- Increment candidate epochs via `MutationObserver` + `IntersectionObserver` (already wired in v17.15)
- Start YouTube-only; defer general-web semantic parsing
- Estimated effort: 1–4 weeks; main risk is CDP Accessibility tree query cost on YouTube's heavy DOM

**Do NOT pick this if** lag dominates — better targets don't help if frames are dropping.

### Candidate C — Single overlay cursor

**Pick this if** v17.15/v17.16 telemetry + observation shows:
- Boundary handoff (main-app ↔ BrowserView) blink or stutter is a top-3 complaint
- Two cursors briefly visible simultaneously despite v17.15's `gcHide()` discipline
- Cursor "jumps" on transitions between main app and BrowserView

**Scope sketch (if picked):**
- Move cursor rendering out of both React content and BrowserView DOM into a transparent click-through overlay window owned by main process
- BrowserView preload performs hit testing, posts anchors to main; main draws one cursor
- Crossfade across surface boundaries (100–150 ms, no dwell accumulation during transition)
- Estimated effort: 1–4 weeks; main risk is OS-level click-through behaviour on Windows display configurations
- **Note:** the audit found that today the codebase only renders **one** cursor at a time — the symptom is abrupt hide/show, not literal dual cursors. If boundary handoff isn't a real complaint after v17.16, **skip this candidate**.

## Anti-candidates (do not pick)

- Replacing the main-app GazeCursor or any Python conditioner — out of scope for the entire v17.x web initiative.
- Native OS cursor warping (OptiKey-style) — sacrifices semantic awareness; only useful as a diagnostic fallback.
- Electron `<webview>` tag migration — Electron docs explicitly recommend against this; stability risk.
- Custom Chromium compositor layer — months of work, ongoing maintenance burden.
- Transformed YouTube surface (Grid 3 style) — interesting long-term direction (v17.20+) but not next-release scope.

## Decision protocol

1. After v17.16 is merged and the patient has used the app for ≥ 1 week, dump telemetry from at least 3 distinct sessions (different times of day, different durations).
2. Compute the metrics in each candidate's "Pick this if" block.
3. If a candidate's signals dominate by > 2× over the other two: pick it.
4. If signals are mixed: prefer Candidate A (cheapest, lowest regression risk) and revisit after another telemetry cycle.
5. If none of the signals are bad: ship nothing, close v17.17.

When the chosen candidate is locked in, replace this file with a real spec matching the v17.15 / v17.16 structure (Phase 0 guardrails, numbered DoD with concrete thresholds, telemetry additions, verification scenarios, numbers block, escalate list).
