## Phase 1 — Definition of Done

The patch is complete only when **all six** items below are implemented and verified. Each item lists a concrete numeric threshold; pick exactly the value stated unless you find evidence it's wrong, in which case flag it for review rather than silently changing it.

### DoD-1. Candidate-epoch tracking + stale-belief invalidation

Add a single integer `state.candidateEpoch` to the injected script. Increment it on any of:

1. **Scroll** of the document, `window`, or any scrolling ancestor of the gaze region. Use a `scroll` listener on `window` with `{ passive: true, capture: true }` and a 100 ms throttle (leading edge fires immediately, trailing fires at end).
2. **Resize** of the window or BrowserView (`resize` event on `window`).
3. **YouTube SPA route change.** YouTube fires `yt-navigate-finish` on `document`. Listen for it. Also listen for `popstate` and `hashchange` as a fallback.
4. **DOM mutation affecting candidates.** Use a single `MutationObserver` observing `document.body` with `{ childList: true, subtree: true, attributes: true, attributeFilter: ['style', 'class', 'hidden', 'aria-hidden'] }`. Batch mutations: process at most once per animation frame using `requestAnimationFrame`. Only increment epoch if at least one mutation touches an element matching the candidate selector list.
5. **Candidate list churn.** When the per-frame candidate query result differs in size or in any element identity vs. the previous frame, increment the epoch.

**On epoch increment, apply posterior decay:**
- **Soft epoch** (scroll, single candidate added/removed): multiply every entry in `state.cardPosteriors` by `0.30`.
- **Hard epoch** (route change, ≥5 candidates changed in one batch, or resize): clear `state.cardPosteriors` entirely.

**Done when:** scrolling through a YouTube results page during dwell visibly resets the dwell ring within ~100 ms, and the posterior never re-locks to a stale candidate whose rect has moved off-screen.

### DoD-2. Snap to visible thumbnail region, not full renderer rect

For compact / sidebar-class candidates, replace `safeRect(card)` in the snap-distance calculation with a clamped rect.

**Which selectors are "compact":**
- `ytd-compact-video-renderer`
- `ytd-compact-playlist-renderer`
- `ytd-compact-radio-renderer`
- `ytd-playlist-panel-video-renderer`

**Clamp algorithm:**
1. Inside the card, query for `a#thumbnail`, then `#thumbnail`, then `ytd-thumbnail`, in that order. Take the first non-null match's `getBoundingClientRect()` as the **thumbnail rect**.
2. Query for `#video-title`, `h3`, `[id="video-title"]` in order — take the first match's rect as the **title rect**.
3. The **snap rect** is the union of thumbnail rect and title rect, clipped to the card's own rect. If neither exists, fall back to clamping the card rect's width to `min(cardRect.width, 0.6 × cardRect.width + cardRect.left)` — i.e. drop the right 40% of the card.
4. Apply this snap rect everywhere the card's rect is currently used for distance computation in `bayesianYoutubeCard()` and `nearestYoutubeCard()`. Continue to use the full card rect for hit-test bounds (the `youtubeCardUnsnapPx: 230` hold zone) — only the **snap target** changes.

**Done when:** dwelling on a compact sidebar card no longer pulls the cursor visual to the far right edge of the card; the cursor anchors at the thumbnail/title center.

### DoD-3. Stable-winner requirement before onset

Add two fields to `state`: `winnerStableId: string | null`, `winnerStableCount: number`.

Per frame, after computing the Bayesian posterior:
1. Identify the top-1 candidate (`winner`) and top-2 candidate (`second`).
2. Compute `margin = winner.posterior - (second?.posterior ?? 0)`.
3. **If** `winner.id === state.winnerStableId`: increment `state.winnerStableCount`.
4. **Else:** set `state.winnerStableId = winner.id`, `state.winnerStableCount = 1`. **Log a `targetSwitch` event** (see telemetry below).
5. **Onset is allowed to start** only when both:
   - `state.winnerStableCount >= 4` (≈ 133 ms at 30 Hz)
   - `margin >= 0.10`

If onset is already running and these conditions cease to hold, see DoD-4.

**Done when:** flicking gaze across a dense card grid no longer immediately starts the dwell ring on whichever card the cursor crosses; the ring only appears after the user actually fixates.

### DoD-4. Cancel onset on candidate movement

During the onset window (between onset start and dwell commit), each frame:
1. Re-fetch the active candidate's current `getBoundingClientRect()`.
2. Compare new center `(cx', cy')` to the center recorded when onset started `(cx₀, cy₀)`.
3. **Cancel onset if** either condition is true:
   - `hypot(cx' - cx₀, cy' - cy₀) > 24` (pixels)
   - The shift exceeds `0.30 × min(rect.width, rect.height)`

On cancel: reset `state.start`, hide the dwell ring, set `state.dwellState = 'idle'`, and log a `dwellCancel` event with `reason: 'candidate_moved'`.

**Done when:** if YouTube re-flows the result list during dwell onset (lazy-loaded thumbnail, ad insertion), the dwell ring disappears rather than tracking the moved card to a wrong final click.

### DoD-5. Per-frame and per-event telemetry

Extend `window.__gcTelemetry` with two ring buffers. **Do not remove** the existing per-click `events()` / `snapshot()` / `posteriors()` APIs — those are still used.

**Frame ring (`window.__gcTelemetry.frames`):**
- Capacity: **4000 entries** (≈ 2.2 minutes at 30 Hz). Overwrite oldest on push.
- Written: every gaze frame, after dwell logic runs.
- Schema (use these exact short field names to keep the buffer small):

| Field | Type | Notes |
|---|---|---|
| `t` | number | ms since session start (use `performance.now()`) |
| `ageMs` | number | wall-clock ms since this gaze sample's backend timestamp; if unknown, `-1` |
| `dtMs` | number | ms since previous frame entry; `0` on first frame |
| `gx`, `gy` | number | raw gaze coordinate in BrowserView-local px |
| `cx`, `cy` | number | rendered cursor coordinate (may differ from gx/gy after snap) |
| `candId` | string \| null | active dwell candidate id (see id scheme below) |
| `candType` | string \| null | one of `'youtube_card'`, `'youtube_skip'`, `'youtube_anchor'`, `'interactive'` |
| `winId` | string \| null | current Bayesian posterior winner id |
| `winMargin` | number | `winner.posterior - second.posterior`; `0` if no second |
| `snapDist` | number | distance in px from gaze to active candidate's snap-rect center; `-1` if none |
| `dwellState` | string | one of `'idle' \| 'onset' \| 'dwell' \| 'commit' \| 'cooldown'` |
| `epoch` | number | current `state.candidateEpoch` value |

**Event ring (`window.__gcTelemetry.events2`):**
- Note: the existing `events()` is per-*click*. Add a separate per-*event* buffer named `events2` to avoid breaking existing callers.
- Capacity: **500 entries**. Overwrite oldest on push.
- Written: on the specific event types below. Each entry has `{ t, kind, ...payload }`.

| `kind` | Payload |
|---|---|
| `targetSwitch` | `{ fromId, toId, fromType, toType, gaze: {x,y} }` |
| `onsetStart` | `{ candId, candType, rect: {l,t,w,h}, posterior, margin }` |
| `onsetCancel` | `{ candId, reason: 'candidate_moved' \| 'gaze_left' \| 'winner_changed' \| 'epoch_change', shiftPx? }` |
| `dwellCommit` | `{ candId, candType, dwellToCommitMs, finalGaze: {x,y}, residual: {dx,dy,mag} }` |
| `routeChange` | `{ url }` |
| `mutationBatch` | `{ size, hardEpoch: boolean }` |
| `trackingLost` | `{ gapMs }` — emitted if `dtMs > 100` |

**Candidate id scheme:** use a stable string built once when a candidate first enters the posterior set: `${candType}:${rect.left|0},${rect.top|0}:${rect.width|0}x${rect.height|0}` rounded to 24 px to match the existing `cardPosteriorKey` rounding. Persist the id across frames if the rect is within the rounding bin.

**Dump API:** add `window.__gcTelemetry.dump()` returning a JSON-serializable object `{ frames: [...], events2: [...], snapshot: {...} }`. The developer copies it from DevTools via `copy(window.__gcTelemetry.dump())`. No new IPC, no new UI button — out of scope.

### DoD-6. No regression in unchanged paths

- The existing `__gcTelemetry.events()` per-click history still works.

## Verification

Run all of these on a 1920×1080 screen with the Tobii Eye Tracker 5 in `start-dev.bat` (real gaze, not simulate). Record the telemetry dump from each scenario for review.

**Scenario A — sidebar snap (the patient's reported failure):**
1. Open YouTube, play any video so the right-side recommendations populate.
2. Look at the *center* of the second recommended video's thumbnail.
3. **Pass:** dwell ring appears centered on the thumbnail; click lands on that video. **Fail:** ring jumps to far-right edge of the recommendations column or to a different card.

**Scenario B — scroll mid-dwell:**
1. On a YouTube results page, start dwelling on a result.
2. Halfway through the dwell, scroll the page (use the existing edge-scroll control).
3. **Pass:** dwell ring disappears within ~100 ms; no click fires. **Fail:** click fires on a stale or relocated card.

**Scenario C — flick across grid:**
1. Sweep gaze quickly across a row of 4–5 video thumbnails.
2. **Pass:** no dwell ring appears on any of them. **Fail:** ring appears and starts counting on the first or second card touched.

**Scenario D — route change:**
1. Click a video. While the new page loads, look at where a thumbnail *was*.
2. **Pass:** no dwell on a stale rect. **Fail:** dwell anchors to old coordinates.

**Scenario E — telemetry sanity:**
1. Run a 60-second session of normal browsing.
2. From DevTools: `t = window.__gcTelemetry.dump()`.
3. **Pass:** `t.frames.length` is roughly 1800 (30 fps × 60 s), `t.events2` contains a mix of `targetSwitch`, `onsetStart`, `dwellCommit`, and per-event `t` timestamps are monotonic. **Fail:** missing fields, NaN, or empty event ring.

**Scenario F — non-YouTube regression:**
1. Browse Google search results, click a link via dwell.
2. **Pass:** identical feel to pre-patch. **Fail:** any change to dwell duration or visual.

## Numbers in one place (so the agent can find them)

| Constant | Value | Where it goes |
|---|---|---|
| Scroll/resize throttle | 100 ms | epoch listener |
| Mutation batch window | 1 animation frame | `MutationObserver` callback |
| Soft-epoch decay | × 0.30 | per-frame posterior multiplier on soft increment |
| Hard-epoch threshold | ≥ 5 candidates changed in one batch | mutation processor |
| Stable-winner frame count | 4 frames (~133 ms @ 30 Hz) | `winnerStableCount` check |
| Stable-winner margin | 0.10 | `winner.posterior - second.posterior` |
| Onset-cancel center shift | 24 px | absolute |
| Onset-cancel relative shift | 0.30 × min(width, height) | relative |
| Frame ring capacity | 4000 entries | `__gcTelemetry.frames` |
| Event ring capacity | 500 entries | `__gcTelemetry.events2` |
| Tracking-loss threshold | dtMs > 100 | `trackingLost` event |
| Sidebar snap width clamp | 0.60 × card width | fallback when no thumbnail rect |

**Do NOT touch these existing constants** in `browserGazeController.ts`:
- `dwellMs: 1100`, `onsetMs: 280`, `postClickCooldownMs: 900`
- `bayesianSigma: 46`, `bayesianAlpha: 0.32`, `bayesianCommitThreshold: 0.45`
- `bayesianOutOfZoneDecay: 0.35`, `bayesianExpandedZone: 1.55`
- `youtubeCardHitZonePx: 130`, `youtubeCardUnsnapPx: 230`

These are calibrated values. Changing them is a separate decision after telemetry data exists.
