Chapter 1

Preface & Scope

GazeConnect Pro is a medical-grade Augmentative and Alternative Communication (AAC) application built for ALS patients whose primary remaining motor channel is eye gaze. This book documents the end-to-end eye tracking pipeline as it exists in build v17.10 — the build that landed after the screen-recording-verified resolution of the “90% dwell abort” bug.

1.1  What this book covers

The book traces a single 30-millisecond infrared sample from the Tobii Eye Tracker 5 device, through six runtime boundaries, through twelve software stages, and into a confirmed HTMLElement.click() call in the React renderer. Each transformation along the way is described both conceptually (what mathematical or perceptual problem it solves) and operationally (which file, which function, which constants).

The scope explicitly includes:

  • The five-layer runtime architecture and the IPC boundaries between layers.
  • The Python backend pipeline: signal conditioning, the velocity-adaptive One Euro Filter, the OptiKey-style four-zone stabilizer, the Adaptive Kalman, the I-VT classifier, and the backend GravityWell magnetism.
  • The React frontend pipeline: coordinate transformation, weighted moving average, semantic snap, EMA smoothing, the v17 R2 visual anchor, and hit-zone expansion.
  • The dwell selection state machine — the heart of the user-perceived behaviour — including all the v17 refinements that fixed the corner-button loop, the 90% abort, and the visual flicker.
  • The Electron BrowserView injection used for in-page gaze control on YouTube, including the Bayesian card posterior (audit recommendation R8).
  • The R1 telemetry layer added for objective accuracy measurement.

Explicitly out of scope (covered in companion documents):

  • Word prediction and language modelling — see docs/word-prediction-system-complete-guide.md.
  • Floor plan and home-design visualisation — see docs/3D_FLOORPLAN_TECHNICAL_AUDIT.md.
  • Color system and visual design tokens — see docs/color-system-research.md.

1.2  How to read this book

Chapters 2–7 are the “forward path” — they describe each stage in the order a gaze sample traverses them. Read them sequentially to build a mental model of the pipeline. Chapter 8 (The Dwell State Machine) is the most behaviourally important and references all earlier chapters; if you have ten minutes and want to understand only the most subtle parts of the system, read Chapter 8 first.

Chapters 9–13 are reference material: the BrowserView pipeline, the telemetry layer, the configuration constants, and a worked example of a single frame's journey. Chapter 14 is the historical version log explaining what each v17.x revision changed and why.

Convention

Source-file references are written as path/to/file.ext. Inline code is set in monospace such as posRef.current.x. Numbers in italic are tuned constants whose values are also collected in the configuration reference table.

Chapter 2

The pipeline runs across five distinct processes on a single Windows machine, communicating over three protocol boundaries. Understanding the boundaries is essential because each crossing introduces serialisation latency, opportunities for state desynchronisation, and unique data formats.

Each layer has a singular responsibility:

1. Hardware Layer — Tobii Eye Tracker 5
USB-attached infrared illuminator + camera. Captures raw eye images, runs on-device pupil + glint detection, emits a 2-D normalised gaze coordinate (0.0–1.0 in both axes, relative to the screen) at 33 Hz steady-state with momentary peaks to 133 Hz under good tracking conditions.
2. Native Bridge — TobiiGazeHelper (C# / .NET 6.0)
A small managed-code daemon that links against Tobii's Interaction Library / Stream Engine SDK (a closed-source DLL Tobii ships for accessibility integrators). It subscribes to the LightlyFiltered gaze stream, normalises timestamps, and emits one JSON line per sample over a local TCP socket on port 5555.
3. Python Backend — main.py + services/
An asyncio process that consumes the TCP stream, runs the entire mathematical signal-processing pipeline (validity, calibration correction, classifier, filter, magnetism), and rebroadcasts the result over a local WebSocket on port 8765. This process is also the home of word prediction, TTS, fatigue monitoring, and the calibration store.
4. Electron Main Process — electron/main.ts
Owns the operating-system surfaces: the main BrowserWindow hosting the React UI, and the embedded BrowserView used for in-page gaze control on YouTube and other web content. Routes Chromium IPC, performs trusted sendInputEvent mouse synthesis when the user dwells on web content, and manages app-level state (Alert Mode, gaze toggle, screen identifier).
5. Renderer — React 18 + TypeScript
Where the user-visible behaviour lives. Receives gaze samples from Python via the renderer-side WebSocket, applies the frontend pipeline (3-tap MA, snap, EMA, R2 anchor), runs the dwell state machine in an animation-frame loop, and translates dwell-completion into HTMLElement.click() calls. Hosts the cursor, dwell ring, highlight rectangle, and all interactive widgets.

Total end-to-end latency from photons hitting the retina to a pixel-level cursor move on screen is the sum of:

Key observation

The mathematical pipeline itself contributes under one millisecond per sample. Optimising the math is not the path to lower latency; the dominant cost is the 33 Hz sensor period (30 ms) and the 16 ms compositor frame. This is why the audit's recommendation R14 (Rust core migration) is justified by IPC reduction, not raw FLOPS.

Chapter 3

The Hardware Layer

The Tobii Eye Tracker 5 is a consumer-grade gaming/Windows-Hello peripheral that Tobii also positions for accessibility hobbyists. It is not the same product tier as Tobii Dynavox PCEye 5 or Tobii Pro Nano, both of which target ALS production deployments. Knowing this distinction is critical to interpreting the residual accuracy observed in clinical use.

3.2  Empirical noise model on a 23″ 1080p display at 60 cm

At our reference deployment (23″ display, 1920×1080, 60 cm viewing distance, ~0.265 mm pixel pitch), 1° of visual angle equals 39.5 px. Tobii does not publish ET5 accuracy maps, but empirical observation during our v17 debugging session showed:

  • Centre-of-screen fixation: ~10–20 px noise envelope (≈0.3°–0.5°)
  • Mid-screen fixation: ~15–30 px noise envelope
  • Corner / edge fixation: ~50–100 px noise envelope (extreme angles cause iris occlusion)
  • Frame gap distribution: 30 ms typical, with routine 100–500 ms drops, occasional multi-second tracking loss during blinks or look-aways.

This noise floor is the dominant constraint on perceived accuracy. The entire frontend pipeline can be viewed as a noise-floor adaptation strategy: the more noise the tracker produces, the more the snap, anchor, and stability mechanisms have to do.

Implication for tuning

Any constant in the pipeline whose units are pixels (snap radius, dwell stability radius, lock-break distance, hit zone expansion) must be at least as large as the worst-case noise envelope at the screen region where it will be used. The v17 series adjusted nearly every such constant after observing edge-region noise of 50–100 px in clinical recordings.

Chapter 4

The Native Bridge (C# / .NET 6.0)

Tobii's Interaction Library (the Tobii.Interaction NuGet package) is a managed-code wrapper over the Stream Engine native DLL. It is .NET-only, which is why this layer exists — Python cannot directly subscribe to Tobii's gaze stream on Windows.

4.1  TobiiGazeHelper

tobii-helper/TobiiGazeHelper/Program.cs

A small .NET 6 console application. On startup it:

  1. Calls SetProcessDpiAwareness so it reports pixels in device coordinates.
  2. Initialises a Tobii Host instance (this loads the Stream Engine DLL).
  3. Creates a GazePointDataStream with the LightlyFiltered profile. Lightly filtered is preferred because heavy in-SDK filtering would conflict with our One Euro + Kalman pipeline downstream — we want the rawest reasonable stream.
  4. Opens a TCP listening socket on 127.0.0.1:5555 and waits for the Python backend to connect.
  5. On each gaze callback, computes screen-pixel coordinates, normalises to 0–1, attaches a high-resolution timestamp, and writes one JSON line: {"x":0.483,"y":0.612,"t":1778523914.401,"valid":true}.

The helper also instruments quality: every 60 frames it emits a [GAZE DEBUG] log line with min/max gap metrics, and a [QUALITY] Frame gap: Nms warning whenever the inter-frame interval exceeds 60 ms. These logs are the canonical source of truth for diagnosing “cursor disappeared” complaints.

4.2  TCP protocol contract

Newline-delimited JSON, one frame per line, ASCII encoding. Backpressure handling is naive: if the Python side is slow, the OS socket buffer fills and the C# side blocks briefly. This is acceptable because the gaze stream is ~33 messages/second and each message is <100 bytes.

Reconnection semantics: if Python disconnects (e.g., backend restart), the C# helper logs [TCP] Disconnected and re-enters the accept() loop. Python's tobii_connector.py retries with exponential backoff. In practice the helper and the backend are launched together by start-dev.bat and stay paired for the lifetime of the Electron process.

5.1  SignalConditioner

python/services/signal_conditioner.py

Purely defensive layer. Does not produce a smoothed signal — only labels samples and synthesises fallback values when the tracker mis-reports. Five jobs:

Validity
Drops samples flagged valid: false by the tracker, or with NaN coordinates.
Blink hold
If a stream of invalid samples lasts ≤ 150 ms (BLINK_HOLD_MAX), the last valid sample is replayed. This prevents a normal blink from being interpreted as a saccade.
Tracking loss
If the invalid stream exceeds 500 ms (TRACKING_LOSS_THRESHOLD), the conditioner emits a gaze_lost event over the WebSocket. The renderer listens and uses this to pause UI animations.
Bounds clamp
Samples are clamped to [0, 1] on both axes (Tobii occasionally emits 0.999 or 1.001 at exact screen edges due to floating-point rounding).
Frozen detection
If displacement < 0.0001 for > 200 frames (≈6 s at 33 Hz), the sample is labelled FROZEN. This catches driver replays and SDK glitches. The threshold is deliberately sub-pixel — real human gaze has natural microsaccade amplitude above this floor.

5.2  Calibration correction

python/services/calibration.py

If the user has run the in-app 9-point calibration wizard, a per-point residual map is stored to disk. The runtime correction applies either:

  • A second-order polynomial fit across the 9 calibration points (default), or
  • A thin-plate spline interpolation (experimental, gated behind a feature flag).

If no calibration is on file, this stage is a no-op pass-through. The renderer-side setGazeOffset in src/utils/gazeSnapping.ts is layered on top as a user-tunable manual offset (the “nudge” setting in Settings → Gaze).

5.3  GazeFilter2D — the One Euro Filter

python/services/one_euro_filter.py

Reference: Casiez, Roussel & Vogel, CHI 2012, 1€ Filter.

The One Euro filter is a velocity-adaptive low-pass — it cleans noise at low gaze velocities (when the user is fixating) and steps out of the way at high velocities (when the user is saccading). Mathematically:

cutoff = min_cutoff + beta * |velocity|
alpha  = 1 / (1 + tau / dt)   where tau = 1 / (2π · cutoff)
x_filt = x_filt_prev + alpha · (x_raw - x_filt_prev)

Per-state cutoffs (set adaptively from the classifier output):

Statemin_cutoffbetaBehaviour
fixation0.40.007Heavy smoothing — tiny tremor suppressed.
saccade3.08.0Near pass-through — cursor jumps with gaze.
glissade0.80.3Moderate — handles post-saccade settling.

The BALANCED preset (min_cutoff=1.0, beta=15.0) is the global fallback when the classifier reports a state not in the above table. The accuracy audit calls out beta=15.0 as unusually high; see version history for the audit response.

5.4  OptiKey pipeline & Adaptive Kalman

python/main.py

The flag USE_OPTIKEY_PIPELINE = True (line ~867) routes samples through a more elaborate stack:

flowchart LR
  classDef stage fill:#eef1f7,stroke:#2c4a7c,stroke-width:1.5px

  C[GazeClassifier]:::stage --> K[AdaptiveKalmanFilter
R-matrix per state]:::stage
  K --> Z[OptiKeyGazeFilter
4 zones: lock / fixation /
edge / free]:::stage
  Z --> M[_apply_magnetism]:::stage

The four-zone OptiKeyGazeFilter applies different damping multipliers depending on the cursor's recent displacement:

  • Lock zone (distance < 0.008 normalised, ~8 px): multiplier 0 — cursor truly stationary.
  • Fixation zone (8–40 px): power-law damping for smooth deceleration.
  • Edge zone (within 6% of viewport edge): multiplier capped low so cursor can reach the edge but doesn't oscillate.
  • Free zone (everything else): multiplier ~0.6–0.9 — fast tracking.

The hysteresis between zones is asymmetric per the Tobii Dynavox patent US10,890,967 (snap-with-different-unsnap-threshold). Lock entry is at 8 px; unsnap at ~14 px.

5.5  GazeClassifier — I-VT

python/services/gaze_classifier.py

Reference: Salvucci & Goldberg, ETRA 2000, Identifying fixations and saccades in eye-tracking protocols.

The classifier emits one of fixation / saccade / glissade per sample. The decision is purely velocity-based plus an N-of-M hysteresis:

velocity = median over VELOCITY_WINDOW=5 samples of dθ/dt  (degrees per second)

if velocity > SACCADE_THRESHOLD (150 °/s):
    high_count++
    if high_count >= SACCADE_ONSET_COUNT (2): state = SACCADE
elif velocity < FIXATION_THRESHOLD (65 °/s):
    low_count++
    if low_count >= FIXATION_ONSET_COUNT (4): state = FIXATION
else:
    state = GLISSADE  (transitioning)

The asymmetric hysteresis (2-of-N for saccades, 4-of-N for fixations) is intentional — the system commits faster to a saccade (so the cursor stays responsive) than to a fixation (so a noisy frame doesn't falsely lock during a saccade).

Audit caveat

The accuracy audit recommends adaptive thresholding (Nyström & Holmqvist BRM 2010 / REMoDNaV BRM 2021) rather than fixed thresholds. The current 65 / 150 °/s defaults are higher than Tobii's own 30 °/s default to accommodate ALS ocular tremor. This is a defensible tuning but not a defensible architectural decision — adaptive thresholding is on the audit roadmap as R5.

5.6  Backend magnetism — GravityWell

python/services/one_euro_filter.py · python/main.py

The GravityWell class is a three-zone gravity model from MAGIC pointing (Zhai, Morimoto & Ihde, CHI 1999). It is invoked as the fallback backend filter when USE_OPTIKEY_PIPELINE = False. When the OptiKey pipeline is active, magnetism is applied by _apply_magnetism in main.py instead, which uses per-context parameter sets:

Chapter 6

The WebSocket Transport Layer

Python broadcasts processed gaze frames over a local WebSocket on ws://127.0.0.1:8765. The renderer connects on startup and stays connected for the lifetime of the application.

6.1  Server side

python/main.py

One asyncio task (_broadcast_gaze_loop) runs at the pipeline's natural rate (~33 Hz). It serialises one JSON frame per sample with this shape:

{
  "type": "gaze",
  "x": 0.483,                  // normalised, post-pipeline
  "y": 0.612,
  "raw_x": 0.485,              // pre-pipeline reference
  "raw_y": 0.611,
  "kalman_x": 0.484,
  "kalman_y": 0.612,
  "conditioned_x": 0.485,
  "conditioned_y": 0.611,
  "coord_space": "window",     // "window" (renderer pixels) or "screen" (global)
  "backend_zone": "lock",      // "lock" | "fixation" | "edge" | "free"

  "is_fixation": true,
  "confidence": 0.93,

  "velocity": 12.4,            // °/s

  "calibration_applied": true,

6.2  Client side

src/hooks/useWebSocket.tsx

The WebSocketProvider hook in the renderer is the entry point. Three implementation details matter:

  1. The onmessage handler dispatches to listenersRef.current — a Set of subscriber callbacks. GazeCursor.tsx registers one such callback. This avoids a re-render on every gaze frame.
  2. An auto-reconnect loop with exponential backoff handles transient WebSocket failures. The renderer never crashes from a backend restart.
  3. The set_screen_size message is sent to the backend every two seconds with the current window's inner dimensions and screen position. This is how the backend knows how to convert screen-normalised gaze to window-normalised gaze when the app is windowed (not fullscreen).

Chapter 7

The Electron Main Process

Electron's main process is responsible for OS-level surfaces. For the gaze pipeline specifically, two responsibilities matter:

7.1  BrowserView injection for in-page gaze control

electron/main.ts · electron/browser/browserGazeController.ts

When the user enters Web Browsing (and especially YouTube), a Chromium BrowserView is created and overlaid on the renderer. The renderer sends gaze coordinates to the main process via IPC (webview:updateGaze); the main process injects them into the BrowserView's JavaScript context via executeJavaScript, where an injected dwell-detection IIFE (see Chapter 10) renders an in-page cursor and detects dwell on web content.

When the in-page dwell completes, the IIFE returns a trustedClick coordinate. The main process then calls view.webContents.sendInputEvent({ type: 'mouseDown', ... }) followed by mouseUp. This is a trusted input event — Chromium treats it identically to a real OS-level click, so user-gesture-gated APIs (autoplay, fullscreen, popups) succeed.

The main process exposes a small set of webview-related IPC handlers. The renderer-side wrapper is src/hooks/useGazeBrowser.ts.

8.1  WebSocketProvider — the subscription layer

src/hooks/useWebSocket.tsx

The provider sits at the top of the React tree (see src/main.tsx). The critical contract is that subscribeGaze(callback) returns an unsubscribe function and the callback is invoked once per WebSocket gaze message. Consumers (the cursor, individual screens) register independently.

8.2  GazeCursor.tsx — anatomy of the cursor component

src/components/core/GazeCursor.tsx (~1500 LOC, the largest single file in the renderer)

The component is structured around three independent loops:

  1. Gaze data loop (handleGaze): triggered once per WebSocket message. Runs coordinate transform, 3-tap MA, snap, EMA, R2 anchor; updates posRef and lastRawPointRef; calls setX / setY.
  2. Dwell frame loop (dwellFrame via requestAnimationFrame): runs at ~60 Hz, independent of gaze sample rate. Reads posRef and lastRawPointRef; performs hit testing; runs the onset / dwell / lock / click state machine; fires HTMLElement.click() when dwell completes.
  3. Refs collection loop: every ~500 ms, refresh the snap-target list (collectSnapTargets) and keyboard key positions (collectKeyboardKeys) so layout changes don't desync the dwell hit test.

Communication between loops is by ref mutation — no React state in the hot path because state updates trigger renders and we cannot afford 60 renders/second.

8.3  Coordinate transformation

The WebSocket payload's x / y can be in two coordinate spaces depending on coord_space:

  • window (preferred): already normalised to the app's content rect. Multiply by window.innerWidth / window.innerHeight.
  • screen (legacy): normalised to the full screen. Must be re-projected using the window's screen position and size.

The renderer maintains windowBoundsRef — a polled snapshot of {x, y, width, height, screenWidth, screenHeight, isFullscreen, isMaximized} — updated every two seconds by the Electron main process. This is what enables the “screen”-coord-space re-projection.

8.4  Three-tap weighted moving average

The first frontend smoothing step. Weights 0.45 / 0.30 / 0.25 on the current, previous, and previous-previous samples:

smoothedX = rawX * 0.45 + pre1.x * 0.30 + pre2.x * 0.25

This is skipped when the backend reports backendOnKey=true or backendMagnetPx > 0.8 px, because the backend is already actively pulling toward a target and additional frontend smoothing would add directional bias.

The audit (companion research brief in this same docs folder) recommended dropping this MA in favour of a properly tuned One Euro β. It was retained in v17 because removing it re-introduced jitter; the audit's conditions for safe removal include adaptive One Euro β tuning which is gated until R1 telemetry data has been collected.

8.5  Semantic snap

src/utils/gazeSnapping.ts

Snap is a soft attraction toward gaze-enabled buttons. Targets are collected from the DOM by querying [data-gaze="true"], [data-gaze-always="true"], [data-gaze-toggle="true"]. For each target, a center, a snap radius, and a max pull strength are computed based on the target's data-gaze-context attribute.

The near-center proximity boost is the v17 addition. 8.7  R2 visual anchor — the heart of the v17 fix

R2 is the most important change to the renderer in the v17 series. It addresses a fundamental problem: on ET5 at 60 cm, the hardware accuracy floor is 12–20 px (0.3–0.5°). No amount of software smoothing can drive the residual below the hardware noise floor. So the cursor's true position will always wobble by 10–20 px around the target's centre.

R2's insight: decouple perceived accuracy from selection accuracy. Once the dwell state machine has confirmed a dwell target (post-onset), force the rendered cursor and posRef to the target's bounding rect centre, every frame. The selection logic (lock-break, target-change detection) continues to read raw gaze; only the visual representation is anchored.

// GazeCursor.tsx — R2 anchor (excerpt)
if (dwellTargetRef.current) {
  const target = dwellTargetRef.current;
  const rect = target.getBoundingClientRect();
  if (rect.width > 0 && rect.height > 0) {
    posRef.current.x = rect.left + rect.width / 2;
    posRef.current.y = rect.top + rect.height / 2;
    setHighlightRect({ left: rect.left, top: rect.top, width: rect.width, height: rect.height });
  }
}

setX(posRef.current.x);
setY(posRef.current.y);

A complementary mechanism — the onset preview anchor — runs during onset: between 100 ms and 250 ms of stable gaze on an onset candidate, posRef is interpolated toward the candidate's centre with pull strength ramping 0 → 0.30. This makes the cursor visibly home in on a key during onset, killing the "cursor at corner before lock" panic that was reported in earlier builds.

Conceptual significance

This is the cleanest example of the BayesGaze principle (Xu et al., Graphics Interface 2021): the cursor displayed to the user is a statement of belief about where they intend to look, not a transcription of where the tracker says they are looking. The displayed position is the system's best guess at the user's intent — which is exactly what an AAC product should show.

8.8  Hit-zone expansion (keyboard nearest-center)

src/utils/hitZoneExpansion.ts

For keyboard keys specifically, the dwell hit test uses a different strategy than the generic multi-point hit test. Each key's bounding rect is inflated by 55 px (KEYBOARD_SNAP_MARGIN); gaze that falls inside any inflated rect is attributed to the key whose centre is closest to the gaze. This guarantees correct disambiguation when gaze falls in the gap between two adjacent keys.

The 55-px margin was raised from 35 in v17.8 specifically to fix the "A" key loop on edge columns of the keyboard — at the left edge, raw gaze noise routinely exceeded 35 px outside the rect, but is rarely greater than 55 px.

Chapter 9

The Dwell State Machine

This is where user-perceived selection accuracy actually lives. The state machine runs inside dwellFrame (an requestAnimationFrame loop at ~60 Hz) in GazeCursor.tsx. It has five states (idle, onset, dwell, locked, click) and a half-dozen edge-handling refinements added across v17.6–v17.10.

stateDiagram-v2
  [*] --> IDLE
  IDLE --> ONSET: cursor enters interactive element
  ONSET --> IDLE: cursor leaves before 250 ms
  ONSET --> ONSET: cursor stays on same element
  ONSET --> DWELL: 250 ms stable + commit
  DWELL --> LOCKED: progress ≥ 10% (≈ +100 ms)
  LOCKED --> CLICK: progress ≥ 100% (≈ +900 ms)
  LOCKED --> IDLE: lock-break (gated by target rect, v17.10)
  CLICK --> COOLDOWN
  COOLDOWN --> IDLE: 1300 ms passed
  DWELL --> SAVED: gaze leaves element
  SAVED --> DWELL: gaze returns within 1000 ms
(progress resumed)
  SAVED --> IDLE: 1000 ms expiry

Figure 9.1Dwell state machine. The SAVED state and the rect-gated lock-break are v17.6 and v17.10 additions respectively.

9.1  Onset phase — 250 ms anti-spurious filter

Onset prevents "drive-by activations" — your gaze scanning across the keyboard shouldn't trigger every key you pass over. The mechanism is simple: when the cursor first enters an interactive element, the element becomes the onset candidate. The candidate must remain stable for ONSET_DELAY_MS = 250 ms (or ONSET_DELAY_ALWAYS_ACTIVE_MS = 100 ms for emergency / toggle buttons) before the dwell counter starts.

If the cursor leaves the candidate before 250 ms — or if a different element becomes the candidate — onset restarts. This is exactly the loop that v17.8 / v17.9 / v17.10 attacked. The chain of failures that caused the "circle restarts at 25%" bug all live in this phase.

9.2  Onset preview anchor — gentle pull during onset

Once onset has been progressing for > 100 ms, the rendered cursor is interpolated toward the candidate's centre with strength ramping 0 → 0.30 over the remaining 150 ms. By the time onset commits, the cursor is visibly homed in on the candidate. This eliminates the "cursor drifts to corner then suddenly snaps to centre" sensation.

9.3  Dwell phase

Once onset commits, dwellTargetRef.current = clickable and dwellStartTimeRef.current = now. Progress is computed as (now - dwellStartTimeRef) / effectiveDwell. effectiveDwell is normally 1000 ms but can be context-specific (400 ms for emergency, configurable per-button via data-dwell-time).

During this phase the R2 anchor (Chapter 8.7) is actively snapping posRef to the target centre on every gaze frame.

9.4  Lock acquisition — 10% threshold

At 10% of dwell progress (~100 ms after dwell start, ~350 ms after gaze landed), the cursor becomes locked. lockPosRef.current is captured to posRef.current (which is at the target centre by R2). From this point on, the cursor no longer moves with gaze — it stays at the lock position until either the dwell completes or the user clearly looks away (lock-break).

9.5  Lock-break gating — the v17.10 fix

Lock-break is the mechanism by which a user signals "I changed my mind, don't fire this dwell." Pre-v17.10, the gate was a single condition:

if (distFromLock > LOCK_BREAK_DISTANCE) {  // 80 px
  // abort dwell
}

This was triggered by raw-gaze noise during legitimate fixation — Tobii ET5 noise of 50–80 px caused the lock-break to fire even though the user was still looking at the button. The symptom: the dwell ring would fill to ~90 %, then everything would reset to zero without firing the click. The patient called it "the 90% abort" and Claude AI's screen-recording analysis pinpointed it on the L key in "hello" at video timestamp 0:23.6.

The v17.10 fix gates lock-break on a second condition — the raw gaze must also be outside the dwell target's bounding rect (with 45 px tolerance):

// GazeCursor.tsx (v17.10) — lock-break with rect gating
if (isLockedRef.current) {
  const distFromLock = Math.hypot(rawX - lockPosRef.current.x, rawY - lockPosRef.current.y);
  let shouldBreakLock = distFromLock > LOCK_BREAK_DISTANCE;

  if (shouldBreakLock && dwellTargetRef.current?.isConnected) {
    const tRect = dwellTargetRef.current.getBoundingClientRect();
    const LOCK_RECT_TOLERANCE = 45;
    const insideTargetRect = (
      rawX >= tRect.left - LOCK_RECT_TOLERANCE
      && rawX <= tRect.right + LOCK_RECT_TOLERANCE
      && rawY >= tRect.top - LOCK_RECT_TOLERANCE
      && rawY <= tRect.bottom + LOCK_RECT_TOLERANCE
    );
    if (insideTargetRect) shouldBreakLock = false;
  }

  if (shouldBreakLock) { /* abort dwell */ }
  return;
}

Now lock-break requires both the distance condition and the rect condition. If the user is still genuinely looking at the button (raw gaze inside rect + tolerance), the lock holds regardless of how far the noise pushes raw gaze from the lock position. Verified by post-fix video analysis: both Ls in "hello" fire cleanly on first dwell.

9.6  Click firing

At 100% dwell progress, the state machine:

  1. Records a telemetry event (target id, rect, gaze position, residual, timing).
  2. Calls dwellTargetRef.current.click() — a standard DOM click event.
  3. Resets all state: dwellTargetRef, onsetTargetRef, savedDwellRef, isLockedRef, highlightRect, dwellProgress.
  4. Sets lastClickTimeRef = now to start the post-click cooldown.

Cooldown lasts 1000 ms + the per-action cooldownAfterActivation (default 300 ms = 1300 ms total). During cooldown, new dwells cannot start unless the target is a toggle / emergency button.

9.7  SavedDwell & visual continuity — v17.6 Option A

Brief gaze excursions (blinks, ALS tremor) used to make the dwell ring visibly reset to 0% then start over when gaze returned. The patient perceived this as a "loop." Option A decouples the visual ring from the logical dwell state:

  1. When gaze leaves a partially-dwelled target, savedDwellRef captures the current progress and a timestamp.
  2. The dwell ring and highlight rectangle are not reset to zero. They remain visible at their last value.
  3. savedDwellExpiryRef is set to now + 1000 ms (FIXATION_TTL_MS).
  4. If gaze returns to the same element within 1000 ms, the dwell resumes from the saved progress (the visual state already matched).
  5. If 1000 ms expires without return, the visuals are cleared.

The BrowserView equivalent uses state.dwellingExpiryAt with a 600 ms grace window on the .dwelling CSS class.

9.8  Anchored hit test — v17.7 Option B and v17.9 unification

The hit test (which DOM element is under the gaze cursor?) was historically run against raw gaze on keyboard screens (the "v18 fast scanning" optimisation). This caused a subtle but fatal bug: the cursor was rendered at posRef (post-snap, EMA, R2-anchored) — visibly on a key — but the hit test was running against lastRawPointRef, which drifted 30–60 px off the key from gaze noise. The user saw the cursor on K; the hit test thought gaze was off K; clickable = null; loop.

v17.7 made the hit test use posRef when a dwell was active (post-onset). v17.9 generalised this to all phases (pre-onset too). The cursor render and the hit test are now the same point — what you see is what you select.

9.9  Edge-aware sticky tolerance

A safety net for the very rare frame where all five multi-point hit-test points miss the rect. If clickable === null but a dwell or onset target is still set, check whether gaze is within a tolerance of the target's rect; if yes, treat the target as the clickable.

Tolerances are edge-aware. Buttons within 80 px of any viewport edge use wider tolerances because Tobii noise at edges is empirically much larger:

PhaseStandard toleranceEdge-button tolerance
Dwell (post-onset)50 px110 px
Onset (pre-dwell)80 px130 px

Chapter 10

The BrowserView Pipeline

When the user is in Web Browsing (especially YouTube), the renderer-side dwell logic does not drive selection of page content. Instead, an injected IIFE inside the BrowserView's JavaScript context runs its own dwell loop. This is necessary because page content is in a separate process and the renderer's React state can't observe DOM changes on the page.

10.1  In-page cursor injection

electron/browser/browserGazeController.ts

On BrowserView load (after dom-ready), the main process injects a CSS sheet plus a JavaScript IIFE. The CSS defines #gazeconnect-cursor: a 52-px ring that follows gaze. The IIFE creates the cursor div and exposes four functions on window:

  • window.gcUpdateAndPoll(x, y, cursorEnabled) — called by the main process on every gaze frame; updates the cursor position, runs the in-page dwell state machine, returns a click request when dwell completes.
  • window.gcHide() — hide the cursor (called when gaze is outside the BrowserView bounds).
  • window.gcResetDwell() — reset the in-page state machine (called after navigation).
  • window.gcBlockDwell(ms) — suspend dwell-click activity for a window (called after a YouTube command so the page can settle).

The skip-ad command in particular is non-trivial. The injected script searches the page for the YouTube skip-ad button using a strict selector list, validates the rect dimensions (real skip buttons are 50–320 px wide, 20–96 px tall, positioned in the lower-right of the player), and then dispatches a synthetic mouse-event sequence directly on the button:

['pointerover', 'pointerdown', 'mousedown',
 'pointerup', 'mouseup', 'click'].forEach(type => {
  el.dispatchEvent(new MouseEvent(type, { ... }));
});
el.click();  // belt and braces

This bypasses Chromium's hit testing entirely, so an overlapping iframe or shadow-DOM element cannot intercept the click. After dispatching, the script returns blockDwellMs: 1500 to pause the in-page dwell so the user's gaze can't accidentally fire a follow-up click on the now-revealed video (which would otherwise toggle play/pause).

10.3  Bayesian card posterior — audit recommendation R8

YouTube card grids are particularly hard for nearest-distance target selection because of tight spacing and gaze noise. v17.4 added a Bayesian posterior over visible cards:

L_i        = exp(-d_i² / (2σ²))         per-card Gaussian likelihood
P_new_i    = α · L_i + (1-α) · P_old_i  temporal smoothing
P_renorm_i = P_new_i / Σ P_new_j        normalise over candidates
commit     = argmax(P) when max(P) ≥ commitThreshold

Defaults: σ = 46 px, α = 0.32, commitThreshold = 0.45, candidate pool = all visible cards within 1.55× the regular hit zone. Posterior decay rate (1-α) per frame for cards no longer in the candidate set.

The posterior accumulates evidence across frames, so a single noisy gaze sample cannot flip the winning card. The temporal smoothing window is ~7 frames (~230 ms at 30 Hz), which matches the in-page dwell onset of 280 ms.

10.4  Asymmetric snap / unsnap (Tobii US10,890,967)

Cards and the skip-ad button use asymmetric hysteresis: the radius required to acquire a target is smaller than the radius required to release it. This prevents boundary flicker when gaze sits near the snap zone edge.

Chapter 11

Telemetry (R1)

src/utils/gazeTelemetry.ts

The R1 telemetry module is a pure measurement layer added in v17 to support objective tuning. It records every dwell-click event with:

  • Target id, context (data-gaze-context), and screen identifier
  • The target's bounding rect at click time
  • The geometric centre of the target
  • The raw gaze sample at click moment
  • The residual: gaze − centre as (dx, dy, magnitude)
  • Onset-to-click duration and dwell-to-click duration

Aggregates (median residual, MAD, drift vector, mean acquisition time, per-context breakdown) are computed on demand and exposed via window.__gazeTelemetry.snapshot() for live inspection in DevTools. Every 20 clicks the module also prints a one-line summary to the renderer console:

[GazeTelemetry] n=20 medianResidual=8.3px MAD=4.1px drift=(2.0,-5.0)px medianAcq=920ms

The same shape of data is recorded BrowserView-side via window.__gcTelemetry — accessible from the BrowserView's DevTools, which is a separate browser-window context from the main renderer.

Strategic purpose

R1 was the audit's #1 recommendation precisely because every subsequent tuning recommendation (R5 adaptive classifier, R6 continuous self-calibration, R10 ALS preset thresholds) requires objective accuracy numbers from a real session. The telemetry buffer captures enough data after 200–500 real clicks to make those decisions empirically rather than subjectively.

Chapter 14

Version History — the v17 series

The v17 series was a focused multi-week debugging campaign driven by the patient's reports and verified by screen-recording analysis. The full chronological sequence:

v17.1   FIX Keyboard snap revert

Reverted v17.0's KEYBOARD_SNAP_RADIUS bump to v15 baseline (90 px / 0.10 strength). The bump had reintroduced the dual-pull with backend keyboard magnetism, causing inter-key snap jumps. Disabled the new near-center proximity boost for keyboard and prediction-strip contexts via useProximityBoost: false.

v17.2   FIX YouTube back button uses browser history

Previously the YouTube panel's Back button always closed the panel entirely. Now it uses browser.canGoBack / browser.goBack first, only closing the panel when no history remains.

v17.3   NEW Two-phase visual anchor (R2 Phase A + Phase B)

Introduced the R2 visual anchor as a separate concept from the snap. Phase A: hard-snap posRef to dwell target centre post-onset. Phase B: smooth pull during onset (100–250 ms). Decouples perceived accuracy from selection accuracy.

v17.4   NEW Sticky tolerance + Bayesian YouTube posterior

Added sticky tolerance (35–70 px) as a safety net for the multi-point hit test. Introduced Bayesian posterior over YouTube cards (audit R8). The sticky tolerance proved too wide; rolled back in v17.5.

v17.5   FIX Narrow sticky + widen HIT_OFFSET

Sticky tolerance reduced 70 → 35 (safety net only). The real flicker fix is multi-point HIT_OFFSET bumped 35 → 70 when a dwell is active. Bayesian α restored from 0.40 to 0.32 for smoother transitions.

v17.6   NEW Visual continuity (Option A)

Decoupled the dwell-ring visual from the logical dwell state. When gaze briefly leaves a target mid-dwell, the visible ring no longer resets to 0 — it persists at its last value during the existing 1000 ms savedDwellRef grace window. BrowserView equivalent uses a 600 ms grace on the .dwelling CSS class.

v17.7   NEW Anchored hit test post-onset (Option B)

Once a dwell target is committed, the multi-point hit test runs against posRef (the R2-anchored cursor position) instead of raw gaze. R2 keeps posRef at target centre, so the hit test cannot miss.

v17.8   FIX Onset-phase sticky + keyboard margin 55

Extended sticky tolerance to also cover the onset phase (60 px standard, 100 px edge-aware). KEYBOARD_SNAP_MARGIN raised 35 → 55 to fix edge-key (A, Z, P) onset resets. Snap-targets-nearest-center now runs as a fallback on keyboard screen too.

v17.9   FIX Unified hit test on posRef

Removed the v18 special case that used lastRawPointRef for keyboard screen hit testing. Hit test now uses posRef on all screens — cursor render and selection point are the same. Sticky tolerance widened slightly (110 px edge / 50 px standard during dwell; 130 px / 80 px during onset).

v17.10   FIX Lock-break gated by target rect — the 90% abort fix

The single most important fix in the series, identified by Claude AI's frame-by-frame video analysis. Lock-break now requires both (a) raw gaze > 80 px from lockPos and (b) raw gaze outside dwell target's rect + 45 px tolerance. Previously trigger-on-(a)-alone aborted dwells at ~90% progress whenever Tobii ET5 noise pushed raw gaze beyond 80 px from centre during legitimate fixation. Verified: the L-key 90% abort and the 15-second Chest/Nebulization loop both resolve.

Appendix

Glossary

AAC
Augmentative and Alternative Communication. Software/hardware that enables communication for people whose primary motor channels are impaired.
Dwell
A selection method where the user holds their gaze on a target for a fixed duration to confirm the selection. The duration in this app is 1000 ms (the dwell phase) plus 250 ms onset = 1250 ms total from gaze landing to click.
Fixation
A period when the eye is approximately stationary (< 65 °/s in this build). Fixations encode intentional looking and are the events the dwell machine accumulates.
Glissade
A short, low-velocity drift that follows a saccade as the eye settles. Distinct from both fixation and saccade.
I-VT
Identification-by-Velocity Threshold. The simplest and most widely used eye-movement event classifier. Salvucci & Goldberg, ETRA 2000.
Lock
An internal state in the dwell machine, entered at 10% dwell progress. While locked, the cursor freezes at lockPosRef and only a deliberate gaze away (lock-break) cancels the dwell.
Onset
The 250 ms anti-spurious filter that runs before the dwell phase. Prevents drive-by activations during gaze scanning.
posRef
The frontend's authoritative cursor position. A mutable ref shared between the gaze handler and the dwell frame loop. Set by the EMA, modified by R2 anchor, read by the hit test and the render.
R2 visual anchor
The mechanism that snaps posRef to the dwell target's centre on every frame during dwell. Decouples perceived accuracy from selection accuracy.
Saccade
A fast (typically > 150 °/s) ballistic eye movement from one fixation point to another.
SavedDwell
The mechanism that preserves dwell progress when gaze briefly leaves a target. If gaze returns within 1000 ms (FIXATION_TTL_MS), the dwell resumes from the saved progress.
Sticky tolerance
A safety net: when the multi-point hit test misses but a dwell or onset target is still set, the target is held if raw gaze is within a tolerance band of the rect.
Tobii ET5
Tobii Eye Tracker 5 — a consumer-grade gaming peripheral repurposed for accessibility use. 33 Hz gaze data rate; 50–95 cm operating distance.
