# v17.16 — Watch / Gaze Paused Mode (Coding Agent Brief)

**Audience:** the coding agent implementing this patch. You will not have the prior conversation. Read this document end-to-end before touching code.

**Goal in one sentence.** Stop the web cursor from generating accidental clicks while a video is playing, and give the patient a small bilingual control rail to pause gaze, play/pause the video, change volume, and go back — without redesigning the screen.

**Release tag:** `v17.16-watch-mode`
**Branch base:** `v17.15-web-stability` (this patch lands on top of v17.15)
**Scope:** playback-state detection + in-video dwell suppression + an additive Watch Mode control rail. No backend or main-app cursor changes.

**Hard prerequisite:** v17.15 must be merged. This spec assumes the candidate-epoch system, telemetry rings (`__gcTelemetry.frames`, `events2`), and stable-winner logic are already in place.

## Phase 1 — Definition of Done

The patch is complete only when **all ten** items are implemented and verified.

### DoD-1. Playback-state detection (injected)

In the injected script, maintain three new fields on `state`:
- `state.activeVideo: HTMLVideoElement | null`
- `state.videoPlaying: boolean`
- `state.videoRect: { l, t, w, h } | null` — refreshed each frame from `state.activeVideo.getBoundingClientRect()`, null when no video

**Detection rules:**
1. On each candidate epoch increment (DoD-1 of v17.15), re-scan for video elements: pick the largest `<video>` by visible area that's currently rendered (`!== null` rect, width × height > 0). If unchanged from previous, keep the same reference.
2. Bind `play`, `pause`, `ended`, `emptied` listeners on `state.activeVideo`. Unbind from any previous element. Update `state.videoPlaying` from listener events; on `emptied` or null video, set to `false`.
3. Also poll `video.paused` once per second as a sanity check — YouTube sometimes fires events inconsistently across SPA route changes.

**State emission to React:** debounce 100 ms, then send IPC `webview:playbackState` with payload `{ playing: boolean, hasVideo: boolean, rect: { l, t, w, h } | null, fullscreen: boolean }`. The `fullscreen` field is `document.fullscreenElement !== null`.

**Done when:** starting any HTML5 video (YouTube, news embed, any `<video>`) flips `playing` to `true` within 200 ms of `play` event; pausing/ending flips it to `false`.

### DoD-2. In-video dwell suppression

In the per-frame dwell update inside the injected script:

1. Compute `gazeInsideVideo = state.videoPlaying && state.videoRect && pointInRect(gx, gy, state.videoRect)`.
2. If `gazeInsideVideo === true`:
   - Set `state.dwellState = 'suppressed'`.
   - Hide the dwell ring (`#gazeconnect-cursor` keeps the dot, removes the `.dwelling` / `.clicking` classes).
   - Do **not** update `state.cardPosteriors`, do not start onset, do not commit clicks.
   - Continue logging telemetry frames with `dwellState: 'suppressed'`.
3. If gaze leaves the video rect: resume normal dwell processing on the next frame (no cooldown, no transition delay — the user should be able to click an edge control immediately).

**Important:** suppression applies **only** inside `state.videoRect`. The rest of the page (ads, recommendations, toolbar, scrollbar) keeps normal dwell. This is so the patient can still close ads or click "Back" outside the video frame.

**Done when:** during playback, looking anywhere inside the video frame produces no dwell ring; looking at any control outside it (e.g., the recommendations column or the Watch Mode rail) works normally.

### DoD-3. Watch Mode rail UI — auto-show on playback

When React receives `webview:playbackState` with `playing: true` AND `fullscreen: false`:

1. Show a Watch Mode rail anchored to the bottom edge of the WebBrowsingScreen content area.
2. The rail is a horizontal strip **120 px tall** spanning the BrowserView width.
3. Resize the BrowserView's `height` down by 120 px so the rail sits **outside** the BrowserView (Electron BrowserView always paints above HTML — overlays don't work; we must shrink it). Restore full height when the rail is dismissed.
4. The rail starts in **expanded** state on first show.

When `webview:playbackState.playing` flips to `false`: hide the rail and restore full BrowserView height.

**Rail container styling:**
- `background: rgba(13, 17, 23, 0.96)` (matches app dark bg with slight transparency)
- `border-top: 2px solid #2DD4BF` (accent)
- Buttons horizontally centred, evenly spaced with 24 px gap
- All buttons use the existing `GazeButton` component with `dwellCategory="navigationButton"`

**Done when:** playing any video pushes the BrowserView up by 120 px and reveals a control strip at the bottom; pausing the video restores full height.

### DoD-4. Auto-collapse + re-expand

To reduce permanent screen real estate, the rail collapses after inactivity.

1. After **6 s** of no main-app gaze entering the rail's bounding rect, collapse to a single small button.
2. Collapsed state: one **320 × 88 px** button labelled `Show Controls / नियंत्रण दिखाएँ`, anchored bottom-centre. BrowserView height is then only reduced by 88 px (not 120 px). The 88 px floor matches the expanded rail button height and respects the 80 px minimum (2° visual angle at 60 cm) from `CLAUDE.md`.
3. On main-app gaze entering the collapsed button's bounds **or** the previous expanded-rail bounds: re-expand within one frame. Reset the inactivity timer.
4. On dwell-click of `Hide Controls / नियंत्रण छिपाएँ` in the expanded rail: collapse immediately.

**Done when:** the rail visibly retracts to a single button after a few seconds of looking only at the video, and re-expands when the user looks down.

### DoD-5. Bilingual buttons in the expanded rail

Six buttons, left-to-right, each **180 × 88 px** minimum (per CLAUDE.md 80px floor):

| English | Hindi | Action |
|---|---|---|
| `Pause Gaze` | `नज़र विराम` | DoD-7 |
| `Back` | `वापस` | DoD-9 |
| `Play / Pause` | `चलाएँ / रोकें` | DoD-8 |
| `Volume −` | `आवाज़ कम` | DoD-8 |
| `Volume +` | `आवाज़ बढ़ाएँ` | DoD-8 |
| `Hide Controls` | `नियंत्रण छिपाएँ` | DoD-4 collapse |

Two-line button labels: English on top (16 px), Hindi underneath (14 px). Use the existing app font (Inter / Noto Devanagari fallback).

**Important:** the rail **must not** include a Full Screen button. Per DoD-10, fullscreen is auto-exited; there is no user-facing entry point.

**Done when:** all six buttons render with both languages legibly and respond to dwell with the same feel as other navigation buttons in the app, and no Full Screen button exists anywhere in the rail.

### DoD-6. Watch Mode telemetry events

Extend `window.__gcTelemetry.events2` (from v17.15) with these new `kind` values. **Do not add a new buffer.**

| `kind` | Payload |
|---|---|
| `videoStart` | `{ src, durationS }` — fires on `play` event when previously paused/no-video |
| `videoPause` | `{ src, currentTimeS }` |
| `videoEnd` | `{ src }` |
| `dwellSuppressed` | `{ gazeRel: {x,y} }` — emit at 1 Hz max while gaze is in video region and playing; do not spam every frame |
| `watchRailShown` | `{ reason: 'autoOnPlay' \| 'gazeEnter' }` |
| `watchRailCollapsed` | `{ reason: 'autoInactivity' \| 'manualHide' }` |
| `watchActionClicked` | `{ action: 'pauseGaze' \| 'playPause' \| 'back' \| 'volumeUp' \| 'volumeDown' \| 'showControls' \| 'hideControls' }` |
| `gazePaused` | `{ durationS }` |
| `gazeResumed` | `{ reason: 'timeout' \| 'manual' }` |
| `fullscreenEnter` | `{}` — fires when fullscreen detected, before auto-exit attempt |
| `fullscreenAutoExited` | `{ success: boolean, reason?: string }` — fires after `document.exitFullscreen()` resolves/rejects |
| `fullscreenLockoutRisk` | `{ attempts: number }` — fires only if auto-exit fails repeatedly; signals follow-up release is needed |

Frame ring (`__gcTelemetry.frames`) gains no new fields. The `dwellState` field will already record `'suppressed'` per DoD-2.

**Done when:** a 60 s session of video playback + Watch Mode interaction produces a populated event ring covering at least `videoStart`, `watchRailShown`, `watchRailCollapsed`, and one `watchActionClicked`.

### DoD-7. Pause Gaze action

On dwell-click of `Pause Gaze`:

1. Set React state `webGazePaused = true` and start a **60-second countdown**.
2. While paused, the main process **stops calling `view.webContents.executeJavaScript(...)` with gaze coordinates** for the BrowserView (gate the existing handler at `main.ts:1780–1874`). The injected cursor element should hide on the next gaze update (which won't come) — explicitly send one final `gcHide()` call on entering paused state to clear immediately.
3. A countdown chip is rendered in the **top-right** of the WebBrowsingScreen content area showing remaining seconds: `30s / 30 सेकंड`. Size: 140 × 56 px, accent border, monospace digit.
4. Auto-resume at 0 s. Emit `gazeResumed { reason: 'timeout' }`.
5. The rail's `Pause Gaze` button label changes to `Resume Gaze / नज़र फिर शुरू` while paused. Dwell-click on it resumes immediately and emits `gazeResumed { reason: 'manual' }`.
6. **Main-app gaze (outside BrowserView) is unaffected** — the user can still gaze at the rail buttons. The pause only suppresses the *web-side injected* cursor.

**Done when:** dwelling Pause Gaze stops the in-page cursor immediately, the countdown chip ticks down, and the user can still operate the rail with main-app gaze the whole time.

### DoD-8. Play/Pause and Volume actions

Add three new IPC handlers in `electron/main.ts`, exposed via `electron/preload.ts`:

- `webview:playPause` — main process calls `view.webContents.executeJavaScript("document.querySelector('video')?.paused ? document.querySelector('video')?.play() : document.querySelector('video')?.pause()")`.
- `webview:volumeUp` — call `view.webContents.executeJavaScript("const v=document.querySelector('video'); if(v){v.volume=Math.min(1, v.volume+0.1)}")`.
- `webview:volumeDown` — same, with `Math.max(0, v.volume-0.1)`.

These run on dwell-click of the corresponding rail buttons. They're one-shot calls, not per-frame, so `executeJavaScript` is fine here (the v17.17 IPC rewrite addresses only the *continuous gaze stream*).

**Done when:** the Play/Pause button toggles video state, and the volume buttons audibly raise/lower volume in 10% steps.

### DoD-9. Back action

`webview:back` IPC handler calls `view.webContents.goBack()` if `canGoBack()` returns true, else no-op.

**Done when:** dwelling Back navigates the BrowserView one step in history.

### DoD-10. Fullscreen — auto-exit (do not allow lockout)

A fullscreen video would hide the Watch Mode rail and leave the patient with no gaze-accessible recovery path. For an ALS user, that is a lockout: they may not be able to press Esc, the React banner cannot reliably paint above a true fullscreen `<video>`, and in-video dwell suppression would block the YouTube player chrome's own exit affordance. Treat fullscreen as **unsupported in v17.16**: detect, auto-exit, restore the normal rail.

When the injected script's playback poll observes `document.fullscreenElement !== null` AND the active video is playing:

1. Emit `fullscreenEnter {}` immediately.
2. Call `document.exitFullscreen()` from the injected script. This works without a user gesture — the API restriction is only on `requestFullscreen()`. Await the returned promise.
3. On resolve: emit `fullscreenAutoExited { success: true }`. Restore the normal Watch Mode rail at full 120 px height on the next frame.
4. On reject: emit `fullscreenAutoExited { success: false, reason: <error.message> }` and retry up to **2 more times** at 500 ms intervals.
5. After **3 failed attempts within 2 s**: stop retrying, emit `fullscreenLockoutRisk { attempts: 3 }`, and surface a small accent-bordered chip top-right: `Please press Esc / कृपया Esc दबाएँ`. This is the only place the spec falls back to keyboard guidance — and only after auto-exit has provably failed.
6. After successful exit, show a non-blocking bilingual safety chip top-right for **4 s**: `Fullscreen disabled for gaze safety / नज़र सुरक्षा के लिए पूर्ण स्क्रीन बंद है`. Size 320 × 56 px, accent border, dismisses on timeout only (no dwell required).
7. In-video dwell suppression (DoD-2) continues to apply during the brief fullscreen window so no accidental clicks reach the YouTube player chrome.
8. The rail **must not** include a Full Screen button. There is no user-facing entry point into fullscreen mode.

**Done when:** pressing YouTube's player fullscreen button results in a fullscreen state that lasts < 250 ms in the happy path, after which the BrowserView is back to normal Watch Mode rail and the bilingual safety chip is briefly visible. The lockout-risk path is exercisable in a unit/manual test where `exitFullscreen()` is stubbed to reject.

## Verification

Run on the same 1920×1080 + Tobii ET5 setup as v17.15. Save telemetry dumps for each scenario.

**Scenario A — playback triggers Watch Mode (and resize is clean):**
1. Open YouTube, play a video.
2. **Pass:** within 200 ms of `play`, the BrowserView resizes up, the expanded rail appears, and `videoStart` + `watchRailShown` events fire.
3. **Resize-cleanliness check:** during the BrowserView shrink (120 px), there must be no visible flicker, no white flash, and no stale gaze→cursor mapping. Within 100 ms of resize complete, the rendered cursor position relative to the new BrowserView bounds must match actual gaze. If telemetry shows `gx`/`gy` values briefly outside the new bounds, the bounds-sync at `WebBrowsingScreen.tsx:674–693` needs a one-shot override coordinated with the rail-show transition. Escalate if observed.

**Scenario B — in-video dwell is suppressed:**
1. While video plays, look at the centre of the video frame for 5 s.
2. **Pass:** no dwell ring, no click. Telemetry shows `dwellState: 'suppressed'` in `frames` and at least one `dwellSuppressed` event.

**Scenario C — rail collapses and re-expands:**
1. Look only at the video for 8 s.
2. **Pass:** rail collapses to a single 320 × 88 px `Show Controls` button at ~6 s; BrowserView height grows by 32 px (120 − 88).
3. Look down at the collapsed button.
4. **Pass:** rail re-expands within one frame, BrowserView shrinks back to the 120 px reduction.

**Scenario D — Pause Gaze countdown:**
1. Dwell-click `Pause Gaze`.
2. **Pass:** in-page cursor disappears immediately, countdown chip appears in top-right and ticks down from 60 to 0, in-page gaze stays disabled the whole time. Main-app rail buttons still work.
3. At 0 s: cursor resumes, chip disappears, `gazeResumed { reason: 'timeout' }` fires.

**Scenario E — Play/Pause + Volume:**
1. Dwell Play/Pause → video toggles play state.
2. Dwell Volume + three times → audible volume rises ~30%.
3. Dwell Volume − five times → volume falls (clamped at 0).
4. **Pass:** all actions log `watchActionClicked` events with correct `action` values.

**Scenario F — Back:**
1. Click a video on YouTube; once it loads, dwell Back.
2. **Pass:** BrowserView navigates back to the previous page (results list).

**Scenario G — Fullscreen auto-exit (no lockout):**
1. Press YouTube's player fullscreen button.
2. **Pass:** within 250 ms, fullscreen is exited automatically, BrowserView resizes back to Watch Mode rail at 120 px, and the bilingual safety chip `Fullscreen disabled for gaze safety / नज़र सुरक्षा के लिए पूर्ण स्क्रीन बंद है` appears top-right for 4 s. Telemetry shows `fullscreenEnter` immediately followed by `fullscreenAutoExited { success: true }`.
3. **Fail:** fullscreen persists > 500 ms, rail/chip absent, or any rail button exposes a Full Screen entry point.
4. **Lockout-risk path (manual):** stub `document.exitFullscreen` to reject; re-run step 1. **Pass:** after 3 attempts the `Please press Esc / कृपया Esc दबाएँ` chip appears and `fullscreenLockoutRisk { attempts: 3 }` is logged.

**Scenario H — non-video page regression:**
1. Browse Google search, news, knowledge panel.
2. **Pass:** no rail appears anywhere, no BrowserView resize, dwell behaves exactly as v17.15.

**Scenario I — v17.15 regression:**
1. On a YouTube results page (no video playing yet), repeat v17.15 Scenarios A, B, C.
2. **Pass:** all v17.15 scenarios still pass identically.

## Escalate (don't decide silently)

Stop and ask if you find:
- The BrowserView height resize causes flicker or layout race conditions with the existing `useBrowserViewBoundsSync` (lines 674–693). Coordinating the rail height change with the 350 ms bounds-sync interval may need a debounce or one-shot override.
- YouTube fires `play`/`pause` events inconsistently for ads vs. main video. If ad transitions cause rail flicker, we may need a 500 ms hysteresis on the `playing` → `false` transition specifically.
- `<video>` rebinding after SPA route change leaves the rail stuck visible. The 1 Hz sanity poll should catch this, but verify.
- The expanded rail's button text overflows in Hindi at 180 px width. Widen to 220 px and flag.
- Renaming `browserInteractionMode` → `browserPanelMode` (recommended in Phase 0) touches more than ~50 call sites or risks v17.15 regression. Keep the old name in that case; the new state names listed in Phase 0 (`watchRailState`, `webGazePaused`, `videoPlaybackState`) remain mandatory regardless. **Do not** introduce a `watchMode: boolean` substitute.
- `document.exitFullscreen()` rejects on YouTube specifically (e.g., due to a Picture-in-Picture interaction). The 3-attempt retry should surface this via `fullscreenLockoutRisk`; if you see this in real use, escalate before shipping.
